## 1. ゲーム概要

女性向けワンシチュエーションエロゲ。マフィア幹部（イケメン）を緊縛拘束し、快楽堕ちさせる。

キャラクターは「反抗心の高い誇り高いマフィア幹部」であり、嫌なはずなのに興奮が高まり気持ちよくなってしまう困惑・屈服がゲームの見どころ。

## 2. 内部パラメータ

### 2-1. 主要パラメータ

| パラメータ | 値域 | 役割 | 状態遷移上の位置づけ |
|---|---|---|---|
| `Resistance` | 0.0〜1.0 | 拒絶度（高=拒絶、低=受容） | **遷移の横軸**。状態差の主軸その1 |
| `Drive` | 0.0〜1.0 | 理性崩壊の総量 | **遷移の縦軸**。状態差の主軸その2 |
| `Arousal` | 0.0〜1.0 | 快感の反応量 | **遷移タイミングの補助条件**。主軸にしすぎない |
| `Fatigue` | 0.0〜1.0 | 消耗・疲弊 | **End処理の制御軸**。短時間で効きすぎないようにする |
| `EdgeTension` | 0.0〜1.0 | 射精欲・オーガズム欲 | **駆け引きの中心**。Arousalの従属にしすぎない |
| `DriveBias` | -1.0〜1.0 | 崩壊の質（-=トロトロ / +=アヘ顔） | **End分岐と質感演出**。途中状態でも徐々に蓄積 |

### 2-2. 補助パラメータ

| パラメータ | 値域 | 役割 |
|---|---|---|
| `NeedMotion` | 0.0〜1.0 | 腰の反応・求める動き（内部蓄積） |
| `RejectMotion` | 0.0〜1.0 | 拒絶・苛立ちを表現するランダム発火モーション。Resistanceに応じて発火率変化 |
| `RejectHabituation` | 0.0〜1.0 | 安定した刺激には慣れ、急な刺激でRejectMotion発火率を上げる関数 |

### 2-3. 遷移条件で使う参照値

| 参照値 | 説明 | enum値 |
|---|---|---|
| `Arousal` | 主要パラメータ | 0 |
| `Resistance` | 主要パラメータ | 1 |
| `Fatigue` | 主要パラメータ | 2 |
| `Drive` | 主要パラメータ | 3 |
| `OrgasmCount` | 現在の状態での射精回数（状態変化でリセット） | 4 |
| `DriveBias` | 主要パラメータ | 5 |
| `EdgeDwellTime` | エッジモード累積滞在時間（秒）。射精でリセット | **6** |

> ⚠️ `EdgeDwellTime` の enum値は **6**。以前 BrokenDownMode が6を占めていたが廃止済みにより繰り上げ。asset書き込み時は `param: 6` が正しい値。

## 3. 入力定義

### 3-1. MainIntensity（ピストン強度）

- スライダーで制御（0.0〜1.0）。手を離してもその値で停止・継続。
- `isActive` フラグで停止状態と区別。

### 3-2. SubA（チンコ刺激）

- クリック or バイブトグル（ON/OFF）
- `Arousal`増加 / `Resistance`低下 / `DriveBias+`方向 / `Drive`増加の傾向

### 3-3. SubB（乳首刺激）

- クリック or バイブトグル（ON/OFF）
- `Arousal`増加 / `Resistance`増加（デフォルト） / `DriveBias-`方向 / `Drive`増加の傾向
- `SubBResistanceChange` を負値に設定すれば状態ごとに低下に変更可能

### 3-4. SubA + SubB 同時入力

- SubA/Bはそれぞれ独立して加算
- `DriveBias`のみ例外：同時ON時は合算値の絶対値をマイナス方向に強制 `(-|SubAGain + SubBGain|)`

### 3-5. Band（入力帯）判定

| Band | 条件 |
|---|---|
| `Stop` | `isActive == false` |
| `Below` | `MainIntensity < TolLow` |
| `Within` | `TolLow <= MainIntensity <= TolHigh` |
| `Above` | `MainIntensity > TolHigh` |

`TolLow` / `TolHigh` は状態ごとに異なる（後述）。

### 一覧

## 7. 設計上の重要方針

### 避けたい挙動
- どの状態も「Arousal が高いか低いか」だけで差が決まる
- Drive が低いのに終盤状態っぽく見える
- `BrokenDown` が単に `Surrendered` の上位互換にしか見えない
- `EdgeTension` が `Arousal` のおまけになり、駆け引きがなくなる

### 重点的に差を出したい組み合わせ
- **Defensive vs Acclimating**：どちらも感じうるが、Defensive は拒絶と反発、Acclimating は受け入れと順応。差は Resistance/Drive で出す。
- **Surrendered vs BrokenDown**：どちらも高受容だが、Surrendered は理性が残っている。BrokenDown は理性が切れて DriveBias 差が強く出る。
- **Overridden vs FrustratedCraving**：Overridden は「押し切られ」、Frustrated は「欲求不満で求めてしまう」。

### OrgasmCount をマイルストーンとして使う設計

- `Defensive → Overridden`：Defensive 状態中に1回射精が必要（押し切る難しさの表現）
- `Acclimating → Surrendered`：Acclimating 中に1回射精が必要（駆け引きを挟む）
- `Surrendered → BrokenDown`：Surrendered 中に3回射精が必要（最終到達の重さ）
- `Frustrated → BrokenDown`：Frustrated 中に2回射精が必要（焦らし蓄積の表現）

### Surrendered → BrokenDown の分岐設計

プレイヤーの操作スタイルと DriveBias の蓄積によってエンド方向が変わる：

```
優しい操作（Below継続）+ DriveBias < 0  →  [トロトロ] BrokenDown → End_C_White
激しい操作（Above継続）+ DriveBias >= 0  →  [アヘ顔]  BrokenDown → End_C_Overload
```

## 8. 実装上の注意点

### ルール評価順の重要性
`StateTransitionConfig.asset` のルールは **上から順に評価**し最初にマッチしたものが適用される。
各状態内の評価順：**End条件 → 上位遷移 → 横移動 → 後退** の順で記述する。

### ConditionParam の enum 値
`BrokenDownMode` は廃止済みのため削除済み。`EdgeDwellTime` は **6番**に繰り上がっている。
asset に手動で条件を書く場合は `param: 6` = EdgeDwellTime として扱う。

### forceStateLockTimer
`ForceState()` / `ForceStateWithPreset()` 呼び出し後は **1.5秒間** StateResolver の評価をスキップする。
デバッグ強制遷移後に即時他状態へ戻る問題を防ぐためのロック。

### 9-1. Shared の基準値案

まず shared 側で「全状態の最低限の個性」を決め、各状態では差を出したい箇所だけ override する。

#### Main 入力

| Band | Arousal | Resistance | Fatigue | Drive | DriveBias | NeedMotion | 意図 |
|---|---:|---:|---:|---:|---:|---:|---|
| Stop | -0.04 | -0.015 | -0.06 | -0.015 | Decay 0.008 | -0.08 | 何もしなければ全体に落ち着く基準形 |
| Below | +0.025 | -0.03 | -0.005 | +0.035 | Shift to `-` 0.025 | +0.08 | 焦らしで受容と欲求が進む基準形 |

### 9-3. この数値案で狙っている挙動

- `Guarded` は `Above` で拒絶が強く立ち、`Below + Sub` でしか懐柔しにくい
- `Defensive` は感じても受容せず、`Above` は押し切りルート、`Below` は焦らしルートに分かれる
- `Overridden` は「押し切られ感」を維持しつつ、`Below` で欲求不満へ横滑りしやすい
- `FrustratedCraving` は `Below` で最も `Drive` と `NeedMotion` が伸び、`Stop` でも少し求めてしまう
- `Acclimating` は `Within` が一番気持ちよく、`Stop` でも続きを欲しがる
- `Surrendered` は `DriveBias` の蓄積が強く出て、`Below / Above` の遊び分けと `Stop` 時の余韻が効く
- `BrokenDown` は `Fatigue` を溜めすぎず、止めても欲しがるサービスタイムを作る

### 9-3.1 Stop 時 NeedMotion 微増の扱い

- `FrustratedCraving / Acclimating / Surrendered / BrokenDown` では、`Stop` 中でも `NeedMotion` を微増させる
- ただし実装上は、`Arousal >= NeedMotionArousalThreshold` または `Sub` 入力ありのときだけ `NeedMotionStopArousedChange` が使われる
- `Arousal` が閾値未満まで落ちたら `NeedMotionStopCalmChange` に切り替わるため、永続的な張り付きにはなりにくい
- それでも張り付きが強い場合は、まず `NeedMotionArousalThreshold` を `0.50 -> 0.55` へ上げるか、各状態の `Stop` 時 `Arousal` 減衰を少し強める

### 9-4. 実装時の優先順

1. `SimSharedConfig.asset` をこの章の shared 基準値へ更新する
2. `Guarded / Defensive / Frustrated / Acclimating` の override を先に反映する
3. `Surrendered / BrokenDown` の `FatigueMultiplier` と `DriveBias` を反映する
4. 最後に Edge 系を調整して、`EdgeTension` の我慢感を詰める
